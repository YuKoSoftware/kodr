**Bug 6 — const &BridgeStruct passes by value instead of by pointer:**
The root cause is in `annotateCallCoercions` (line 498): `is_direct_call` is false for field_expr calls like `renderer.draw(mesh, matrix)`. The const auto-borrow `value_to_const_ref` coercion is never applied.

The fix must be TARGETED to bridge function calls only — do NOT remove the general cross-module exclusion (which prevents double-borrow for regular struct methods).

Add a new code path after the existing `is_direct_call` block (after line 520, before the closing brace of the for loop). The logic:

1. If `!is_direct_call` AND the callee is a `field_expr`:
   - Get the field name (method name) from `c.callee.field_expr.field`
   - Get the object from `c.callee.field_expr.obj`
   - Determine if this is a bridge struct method call by checking if the object's type is a bridge struct (look up in `self.decls.bridge_structs` or check `all_decls` for bridge structs)
   - If it IS a bridge call, look up the bridge function signature to find which params are `const &T`
   - For those params, if the arg is a value (not already a reference) AND is a non-primitive type, inject `value_to_const_ref` coercion

2. Key constraint: SKIP the `self` parameter (index 0 in bridge struct methods) — it's already `&T` and coercing would cause double-borrow.

3. To detect `const &` params in bridge signatures: check `sig.params[idx].type_` for pointer-const patterns. The `resolveCallSig` function returns `FuncSig` with param types — check if param type indicates const reference.

The safest approach is: for bridge struct method calls via field_expr, iterate params starting at index 1 (skip self), check if the bridge declaration's param type is `const &T` (look at the AST param node's type annotation), and if so, add `value_to_const_ref` coercion for args that are non-primitive values.
