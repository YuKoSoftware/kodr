**allocator.orh changes (safe fallback for .allocator() calls — addresses checker issue #1):**

0. Add `bridge func allocator()` declarations to each allocator struct in `src/std/allocator.orh`. This ensures sema recognizes `.allocator()` as a valid bridge method call rather than relying on undeclared method passthrough (which is MEDIUM confidence). The return type is not declared (bridge functions can omit it) — Zig handles the actual type. Updated file should read:

```orhon
module allocator

// allocator — memory allocators for Orhon programs
// Import with: import std::allocator
// Each allocator is a proper type with create() and uniform interface.

// SMP — general-purpose, thread-safe allocator (default)

pub bridge struct SMP {
    bridge func create() SMP
    bridge func allocator(self: &SMP) void
    bridge func deinit(self: &SMP) void
}

// Arena — batch allocator, freeAll releases everything at once

pub bridge struct Arena {
    bridge func create() Arena
    bridge func allocator(self: &Arena) void
    bridge func freeAll(self: &Arena) void
    bridge func deinit(self: &Arena) void
}

// Page — OS page allocator, simple and fast

pub bridge struct Page {
    bridge func create() Page
    bridge func allocator(self: &Page) void
}
```

Note: The return type `void` is a placeholder — bridge functions pass the actual Zig return type through. What matters is that sema sees `.allocator()` as a declared bridge method. Check how existing bridge methods like `.create()` handle return types and follow the same pattern for `.allocator()`. If bridge methods use actual return types, use the appropriate type instead of void.
