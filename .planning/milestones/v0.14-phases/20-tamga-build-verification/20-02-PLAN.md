**Bug 8 — Shared @cImport module for cross-module C type identity:**
When multiple modules `#linkC` the same C library (e.g., both tamga_vma and tamga_vk3d link "vulkan"), generate a shared C import wrapper module that both can reference.

Implementation in `buildZigContentMulti`:

1. Before emitting lib targets, scan all targets for `link_libs` overlap. Build a map: `c_lib_name -> list of module_names that use it`. For libraries used by 2+ modules, generate a shared module.

2. For each shared C library (e.g., "vulkan"), generate a shared wrapper `.zig` file in the cache directory. The file content:
```zig
pub const c = @cImport({
    @cInclude("vulkan/vulkan.h");
});
pub usingnamespace c;
```
The header name should be derived from the `#linkC` metadata. For "vulkan", use "vulkan/vulkan.h". For "SDL3", use "SDL3/SDL.h". This mapping needs to be either:
- A convention: library name = include path (e.g., `#linkC "vulkan"` -> `@cInclude("vulkan/vulkan.h")`)
- Or a new metadata directive: `#cInclude "vulkan/vulkan.h"` alongside `#linkC "vulkan"`

The simpler approach: add a `#cInclude "header.h"` metadata directive alongside `#linkC`. The compiler generates the shared module using the `#cInclude` value. If `#cInclude` is not specified, no shared module is generated (backward compatible).

3. In the generated `build.zig`, create the shared module with `b.createModule()` and `addImport` it to all bridge modules that `#linkC` the same library. Name the shared module `{lib_name}_c` (e.g., `vulkan_c`).

4. Each sidecar that currently does `const c = @cImport(...)` should be changed to `const c = @import("vulkan_c")`. This is a TAMGA-SIDE change that will happen in Plan 03 (workaround removal). The compiler just needs to generate and wire the shared module.

Add the `#cInclude` metadata parsing to orhon.peg if needed (or reuse existing metadata patterns).

Add `c_includes` field to `MultiTarget` struct:
```zig
c_includes: []const []const u8 = &.{}, // C headers from #cInclude metadata
```
